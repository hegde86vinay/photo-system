UPLOAD PATH:
  Client → FastAPI → MinIO(HOT) → PostgreSQL → Elasticsearch
  → Return presigned URL

READ PATH (cache hit):
  Client → FastAPI → Redis HIT → return metadata + presigned URL
  → Client downloads directly from MinIO (bypasses backend)

READ PATH (cache miss):
  Client → FastAPI → Redis MISS → PostgreSQL → populate Redis
  → return metadata + presigned URL

STORAGE TIERING (daily cron):
  HOT  → WARM:  not accessed in 365 days  (~40% cheaper)
  WARM → COLD:  not accessed in 1095 days (~80% cheaper)

🗄 PostgreSQL — Table Partitioning

Photos table partitioned by year (2024–2030). At 5.1B rows, queries hit only the relevant year-partition. Archival = DROP TABLE photos_2024 (instant vs. 730M DELETE rows).

⚡ Redis — Cache-Aside Pattern

Read: try cache → miss → DB → populate cache. Write: DB → invalidate cache. 80% cache hit ratio at 100:1 read:write = only 460 DB reads/sec instead of 2,300.

🔎 Elasticsearch — Inverted Index

LIKE '%keyword%' = full table scan at 5B rows. ES inverted index = O(1) lookup. Dual-write pattern: PostgreSQL (source of truth) + ES (search replica, eventual consistency).

🪣 MinIO — Object Storage Tiers

HOT→S3 Standard, WARM→S3-IA (40% cheaper), COLD→Glacier (80% cheaper). Presigned URLs bypass backend bandwidth bottleneck. ~60% cost savings at 1 PB scale.

📊 Capacity Math

2M photos/day × 200KB = 400GB/day. 7 years = ~1 PB. Peak QPS: 2,300 reads/sec. Bandwidth: ~460 MB/s = 3.68 Gbps peak. CDN mandatory.

🔗 Presigned URLs

Client downloads directly from MinIO/S3 — backend not in the bandwidth path. At 460 MB/s peak, this is essential. Signed with secret key, expire in 1 hour.
